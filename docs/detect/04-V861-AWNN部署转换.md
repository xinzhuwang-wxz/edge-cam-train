- 之后**分类部署、级联部署照这个模子套**（框架本就支持 检测 / 分类 / 检测+分类 全流程）；

## 1. 平台事实（本轮实测确认，覆盖旧文档假设）

| 维度 | 旧假设（plan/CLAUDE §1、docs/classify/04）| **V861 实况（本轮确认）** |
|---|---|---|
| 工具链 | ACUITY/pegasus PTQ | **AWNN 工具链**（`awnntools`，Docker 镜像 `awnn:1.0.2`，内含 PPQ 量化器）|
| 板端模型格式 | `.nb`（VIPLite/awnn） | **`_ipu.param` + `_ipu.bin`**（ncnn 衍生容器），AWNN Runtime 加载 |
| 应用核 | ARM Cortex-**A7** | **玄铁 Xuantie RISC-V**（镜像交叉链 `Xuantie-900-gcc-...-musl32`）→「后处理留 CPU」仍成立，只是 RISC-V 不是 ARM |

**本机可跑性（关键）**：宿主是 **arm64 Mac**，AWNN 镜像是 amd64；已验证 `docker run --platform linux/amd64 awnn:1.0.2` 下 `awnntools` **完整运行**（qemu 模拟）。→ 离线全链（build/profile/simulate）**本机可执行**，无需 x86 机器。

## 2. 端到端流程（AWNN 五步）

```

     ──build──▶ _ipu.param/.bin(板端INT8) + .table(校准表) + _opt.*(优化)
```

## 3. 两条转换路线（用户已定：都转，logits 为主）

我们的检测 ONNX 有两个变体（round2 exports）：

| | **logits（主路）** `main_416_fp32_logits.onnx` | **whole（对照）** `feeder_416.onnx` |
|---|---|---|
| 归一化 | 图内**无** → AWNN 预处理 mean/norm 注入（板端可 `use_npu_preprocess` 折进 NPU）| 图内焊入 Sub/Div |
| sigmoid | **无**（出 logits）| 图尾焊入（出概率）|
| 输出 | `output[1,3598,37]` logits | `output_prob[1,3598,37]` 概率 |
| CPU 后处理 | sigmoid(前5类)+DFL+NMS = `decode_nanodet(num_classes=5)` | 只 DFL+NMS |
| AWNN mean/norm | BGR `[103.53,116.28,123.675]` / `[0.017429,0.017507,0.017125]`（=1/std，实测自焊入常数）| `[0,0,0]` / `[1,1,1]`（喂 0-255 BGR）|
| 合规 | ✅ ADR-0007 / rule#2（INT8 图纯 conv）| ❌ sigmoid 进 INT8 图（对照测掉点）|

**为什么主推 logits**：sigmoid 进 INT8 图会饱和、压 **bird 命门置信度**，且 Vivante 可能软回退该算子（ADR-0007 原因）。logits 路配 `use_npu_preprocess=true` 后，app 端**一样喂 0-255 BGR**（便利不输 whole），但把 sigmoid 挡在 INT8 图外。whole 路作消融，正好量化「sigmoid 进图到底掉多少」——回答 ADR-0007 的待验证项。

config：`configs/config_logits.yml`（主）、`configs/config_whole.yml`（对照）。

## 4. 已验证结果（demos 9 图冒烟，2026-07-07）

**logits build 成功，端到端产出板端 INT8 模型**：
- opset 11 直接吃、无需重导；
- **266 op 全部量化，零回退**——LeakyRelu / Concat / MaxPool / depthwise / channel-shuffle(Slice-Reshape-Transpose) 全 NPU 化；Conv+激活、Conv+Add 融合；
- 校准自检 **cos-sim=1.000000 / mse=0 / max_diff=1e-5**；
- 产物 `build_out/logits/`：**`_ipu.param`(63KB)+`_ipu.bin`(1.35MB)** = 板端 INT8 模型（权重 1.35MB，远在 V861 ~10-15MB 克制预算内）+ `.table` 校准表 + `_opt.*`。

→ **「检测能否转成 VS861 格式」＝ 已答 YES**，且算子全绿、模型极小。

## 4.1 真校准 + P5 框级复现 + 头量化定案（/loop，2026-07-07）

**真校准**（282 张 worksite 均衡子集，从租卡机取回，替换 demos）：logits 整网输出 cos-sim **0.9940**（≈冒烟 0.9949）。

**whole vs logits 对照 → sigmoid 无 INT8 代价**：AWNN **自动把 sigmoid 切到 CPU**（连焊入 sigmoid 的 feeder_416，`sng_Sigmoid` 也 Target=CPU）→ 两路 NPU backbone 完全相同（logits cos 0.994），**不存在「sigmoid 进 INT8 图掉命门」**（回答 ADR-0007 待验证项）。whole 最终概率 cos 0.778 是概率空间测量假象、非更差。

**P5 框级复现（`decode_repro.py`）——真质量判据**：用 simulate dump 的 `data_fp32.bin` 喂 onnxruntime 得 FP32 参考、`output.npy` 得 INT8，各自 `decode_nanodet(num_classes=5,conf0.4,nms0.5)` 比框。**cos-sim 是差代理**——发现真缺陷：

- **AWNN 默认全 INT8 + percentile 校准会裁掉 cls 头稀有高峰值** → 丢强检出：清晰松鼠 FP32 0.910 → INT8 **0.328**（掉 3 logit，漏检）；命门鸟 0.958→0.896（守住但被压）。
- **`minmax` 校准更差**（松鼠 0.116）——非校准算法问题。
- **修法＝混合精度：`hybrid_quantization_conf.white_list` 让 `/head/gfl_cls.0~3/Conv` 头保 fp32** → 松鼠救回 **0.458**（过阈）、**框召回 1.000（4/4）/ IoU 0.945 / 零误报 / `_ipu` 仍 1.3MB**（头 conv 极小、fp32 不增体积）。合 ADR-0007「保护头」。

**定案**：**部署主配置 = `config_logits_mixed.yml`（gfl_cls 头 fp32）**。三变体产物 `build_out/{logits(percentile基线),logits_minmax,logits_mixed(主)}`。
边缘目标（ena24 camera-trap、FP32 本身仅 0.52）INT8 仍会掉＝设计外（[[feeder-scale-target-scope]]：观鸟器只看中大目标）。

## 6. 阻塞项 / 盲点（诚实交代）

- **真·上板**（真 INT8 延迟 / fps / NPU层耗时 / 实测内存 / board-vs-sim 位匹配）＝ 需 V861 板子，当前无 → 做到「需要板前的最后一步」，交接包备好。这与项目一贯「上板段是最大工程盲点」一致。
- **量化精度是下界**：demos 9 图校准，真图替换后才是可信数。
- **模拟器 ≠ 真硅**：simulate/profile 是脱板估计，方向性可信；真数看板子。

## 7. 常见坑（AWNN 文档 §7 + 实测）

- config **禁用 Tab**（全空格）；`opset≠11`（尤其<11）会 build 报错 → 重导 opset=11。
- 图片校准需 `color_space/mean/norm/shape` 互相匹配；npy 数据不走预处理。
- simulate 多输出 blob 顺序须按模型内顺序，否则反量化错配（文档 §7.2#4）。
- 不支持算子（TopK/NMS/Expand/ScatterNd 等后处理）→ `cutstartname/cutendname` 切子图，后处理留 CPU（我们本就如此）。
