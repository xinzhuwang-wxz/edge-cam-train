## 2. 板端基础设施：固件是「基座」，模型是「插件」

板子出厂固件是**裸 Tina Linux，没有 NPU 驱动、没有运行时库**——`/dev/nna` 都不存在
（`CONFIG_AW_NNA_SUNXI` / `GALCORE` / `VIP` 全 not set）。所以 **side-load 走不通，必须重编固件**。

### 2.1 构建配方（编译机，Ubuntu）

```bash
source build/envsetup.sh
lunch v861-perf2                      # ★交互，4 个追问，见下
quick_config config_awnn_runtime -f   # ★官方一键开 NPU，必须带 -f
make -j$(nproc) && pack               # → out/v861m3-xxx_linux_perf2_uart0_nor.img
```

| 坑 | 说明 |
|---|---|
| **`lunch` 有 4 个追问，不是 2 个** | 主型号 `V861` / 子型号 `M3-XXX`（128MB DDR3）/ SYS 电压 `0.92V` / **免责声明 `Y`**。少喂第 4 个 → 死循环刷 `Invalid input...`（日志能涨到几百 MB） |
| **`quick_config` 被判定为 internal** | 必须加 **`-f`** 绕过；它把 `CONFIG_AW_NNA_SUNXI=y` 写进 kernel defconfig，把 awnn_sdk/libawnn/libawipubsp 写进 openwrt defconfig |
| **host 缺 GNU awk** | Ubuntu 默认 mawk → OpenWrt 预检 `Error 1`（约 5 min 就挂）。`sudo apt install -y gawk` |
| **`awnn_verify` 会被丢掉** | 它 `depends on awnn_example`，defconfig→.config 时被丢 → **单独交叉编译**（openwrt musl rv32 工具链），不必重编固件 |

### 2.2 NPU 设备节点是 `/dev/nna`，不是 `/dev/galcore`

驱动 `aw_nna_sunxi` 里 `device_create("nna")`。GALCORE/VIP 是另外的驱动，**V861 官方只开 SUNXI**。
上板验证一律看 `/dev/nna`。

### 2.3 烧录

编译机 headless 无烧录工具，`fastboot` 也走不通（u-boot 没编 `CONFIG_USB_FUNCTION_FASTBOOT`）。
→ **Windows PhoenixSuit**，按住板上丝印 **UBOOT** 键进 FEL 模式，整包烧。

**烧固件不会烧坏硬件**——刷 Flash 是软件写；真正能损坏的只有电源接反/短路/过热（跑 NPU 盯
`/sys/class/thermal/thermal_zone0/temp`，实测满载才 35°C）。软件变砖有 BROM FEL 兜底。

> 📦 **通用固件基座** = `deploy/v861_firmware_base/`。**每块新板只烧一次、与模型无关；换模型只 `adb push`。**

## 3. ★ 铁律：模型必须 16 通道对齐、禁止 channel-split

V861 的 VIP9000PICO **按 16 通道分块**。板端 runtime（AWNN 1.5.1）会拒绝：

```
slice_ipu inplace requires the number of channels to be divisible by 16
→ layer 12 splitncnn_1_slice IPU forward error
```

**NanoDet 因此上不了板** —— ShuffleNetV2 backbone 的 channel split 把通道对半分（58 / 116 / 232），
**全不是 16 的倍数**。这不是工具版本问题（离线 1.0.2 模拟器放行、板端 1.5.1 收紧），是硬件事实。

**PP-YOLOE-s 天生适配**：CSPRepResNet 全是标准 Conv + Concat，backbone 无 channel-split → 一次通过。

> 选型时先查这条。**PicoDet 的 ESNet = 增强版 ShuffleNet，会踩同一个坑**；YOLOX（Apache）是备胎。
> 详见飞书报告的「候选清单」章节。

## 4. 四条实测优化结论（都在真板验证过）

### ✅✅ 静态模式（precompiler）：快 5.3 倍 + 内存恒定 —— 最大的一条

之前一直用 `use_static_mode=0`（dynamic）跑，226ms/帧。**dynamic 模式每帧动态重建计算图**
（官方文档：dynamic 是给动态 shape 的模型如 OCR 用的），~180ms 全花在建图上。
我们模型固定 640² —— **本来就该用静态模式**：

| | dynamic（用错的） | **static（正确姿势）** |
|---|---|---|
| 推理耗时 | 226 ms | **43 ms**（恒定，30 帧长跑 43.0~44.6） |
| npuMemory | 18.02 MB，**且每帧涨 4.7MB 不释放** | **14.43 MB 恒定零增长** |
| 连续推理 | ❌ 第 2 帧就 `dma_mem_alloc fail` | ✅ 任意帧数 |
| precompile | — | 一次性 163 ms（进程启动时） |

C API：`config.precompiler_enable = true` + 循环前调一次 `awnn_instance_precompiler`。
静态/动态输出**解码后逐检出全等**（8 张比对，分数一致到小数点后 3 位）。
注意：`awnn_verify` 对拍/逐层 profile 仍需 dynamic（官方 FAQ），**但生产集成一律 static**。

### ✅ SPP → SPPF：白赚 24% 速度，零精度损失

NPU **只支持 5×5 MaxPool**。SPP 的 9×9 / 13×13 会回退 CPU（+44 ms）。
**9×9 ≡ 2 个串联 5×5，13×13 ≡ 3 个串联 5×5**（stride=1 same-padding 下数学等价，实测误差 **0.000e+00 逐位相同**）。

→ CPU 回退 **3 层 → 0 层**，**−24% 耗时**，**不用重训、不掉精度**。

### ❌ 千万别用混合精度（cls 头保 FP32）

**V861 NPU 是 INT8-only**。把 cls 头设成 FP32 → NPU 根本跑不了，**强制回退 CPU**，
80×80 的 head 卷积在 RISC-V 上奇慢，三层 **+282 ms**（直接翻倍）。

⚠️ **离线模拟器里 FP32 是「免费」的，会骗人** —— NanoDet 那轮的结论在真板上完全相反。
INT8 掉的那点分（~0.05）改用**降低置信阈值**补偿（0.50 → **0.45**），免费。

### ❌ 别把输入降到 416

模型是 640 训的。416 会**整类漏检**（other_animal / person 全丢）+ 误检 + 框碎。不重训别降。

## 5. 板端 INT8 ≡ 上游 FP32（等价性验证）

**先说清楚这一节在回答什么。** 检测精度的权威数字是训练阶段测的
（[round3](../../results/detect/round3/round3-实验报告.md)：**部署域 feeder AP50 = 90.5**）。
上板不改变精度，上板要回答的是另一个正交问题：**板子上的 INT8，是不是忠实复现了 PC 上那个 FP32 模型？**

**93 张**喂食器/庭院场景图（Wikimedia Commons，CC0/CC-BY），**两边喂完全相同的输入字节**：

| 指标 | 结果 |
|---|---|
| 分类头（cls）逐输出余弦相似度 | **0.9993 ~ 0.9996** |
| 回归头（reg / DFL）余弦相似度 | 0.9885 ~ 0.9947 |
| 框匹配率（IoU≥0.5 且同类） | **74 / 78 = 94.9%** |
| 匹配框平均 IoU | **0.9728** |
| 匹配框置信度绝对差 | **0.034** |
| 板端 NPU 延迟 | **中位 226.1 ms**（n=93，`use_awnn_profiler=0`） |

**量化几乎无损。** 分类头比回归头稳——DFL 要对 17 个 bin 做 softmax 再积分，本就对量化更敏感。

> 完整产物（含逐图结果、失败案例、CC-BY 署名册）：`results/detect/v861_awnn_ppyoloe/board_eval/`

### 5.1 ⚠️ 两个会污染结论的坑

**① 对拍必须用同一个 JPEG 解码器。** `PIL` 与 `cv2` 的 IDCT 实现不同，同一张图解出来像素**最大差 34/255**。
一边 PIL 一边 cv2，这个解码差异会被算进「量化掉点」。统一走 `cv2.imread`。

**② 测试图必须围绕产品场景选。** 最初混进了刺猬（地栖夜行的欧洲物种，我们的训练数据是北美相机陷阱、
压根没有）、街猫、室内猫、19 世纪版画 —— 混淆矩阵严重失真，得出「other_animal → bird」的**伪发现**
（全是刺猬造成的）。**剔除域外图后 other_animal 是 10/10 = 100%。**

同理，web 上的长焦庭院猫写真上 cat 只有 11%，但那**不是部署性能**——round3 在真部署域（feeder）测的
**cat AP50 = 96.4**。而且 **FP32 是 17%、板端 INT8 是 11%，两边同塌 ⇒ 是域的问题，不是量化的问题**
（这反而是等价性最有力的证据：板子连模型的弱点都忠实复现了）。

### 5.2 输入格式与官方图形库（C 集成用）

- **AWNN 不支持 YUV 输入**：转换工具链 `color_space` 只有 `RGB/BGR/RGBA/BGRA/GRAY`（工具链指南 §4.2.2），
  NPU 的 ImageProcess 层做不了色彩空间转换 → **NV21→RGB 必须在 CPU 做**（融合单遍 640²：~45ms，见 ppyoloe_run）。
- 官方 **`libaw_simpleocv`**（ncnn 风格）可用于板端 C：`awcv_imread/imwrite(JPEG)`、画框、写字。
  ⚠️ 两个坑：**`CV_8UC3 = 3`（通道数，不是 OpenCV 的 16，传 16 段错误）**；底图 BGR 顺序。
  板子固件里没带这个 .so（399KB），需随工具 push。

## 6. ★★ P0 拦路虎（已解决）：连续推理跑不了第 2 帧 → 根因 = 用错了 dynamic 模式

**dynamic mode（`use_static_mode=0`）下，每次 `awnn_instance_inference` 都新分配 IPU blob 内存、且不释放。**

| 一个进程里推几张 | blobMemory | npuMemory | 结果 |
|---|---|---|---|
| **1 张** | 10.60 MB | **18.02 MB** | ✅ 正常 |
| **2 张** | 15.28 MB | **22.71 MB** | ❌ 顶破 20 MB 保留池 → `dma_mem_alloc fail, size=3276800` → `layer 4 Conv.2 IPU forward error` |

进程退出会归还内存（连开两个进程各推 1 张都正常）。→ 离线批跑的唯一安全形态 = **一进程一张**。

> ⚠️ **分配失败后千万别重试。** 继续硬砸会把内核写坏，板子挂到 **adb offline + USB 都不枚举**，
> **必须物理断电重上电**。看到 `DMA_HEAP_IOCTL_ALLOC failed` 立刻停手。（我们踩过。）

**✅ 解法 = 静态模式（§4 第一条）**，2026-07-21 实测收口：43ms/帧、内存恒定、30 帧长跑稳定。
（早期"static 死锁"是 NanoDet 时代的记录——那个模型本身 slice 不兼容，栽的是模型不是模式；
PP-YOLOE 上 static 完全正常。）调大 DTS `size_pool_mem` 不再是刚需，但**级联塞分类器时仍有用**：
现在检测只占 14.43MB，20MB 池剩 5.6MB，MobileNetV2 分类器要 5.2MB——刚好压线，建议还是调大。

### 6.0 新坑：进程 segfault 崩溃会泄漏 NPU carveout

正常退出（走 destroy/deinit）会归还 NPU 内存；**被 SIGSEGV 杀掉不会** —— 之后任何进程
`instance_create` 都 `dma_mem_alloc fail`。恢复：**`adb reboot` 软重启即可**（无需物理断电；
板子本身活着，只是 NPU 池被占满）。

### 6.1 `use_awnn_profiler` 的真实影响（订正）

一度以为 profiler 会多吃内存把池顶破——**错的**。同一张图实测：

| profiler | npu_ms | npuMemorySize |
|---|---|---|
| **0** | **220.60** | **18.0175 MB** |
| **1** | 244.44 | **18.0175 MB**（完全一样） |

**profiler 只加 ~24 ms 耗时，不占一个字节 NPU 内存。** 但部署态仍应关掉（白省 10%）。

> **口径**：早期记的 **238 ms** 是开着 profiler 测的；**部署态真实延迟 = 226 ms**（93 张中位数）。

## 9. 增量实验记录

### 2026-07-21 · autoresearch：性能一致前提下的延迟压缩（4 迭代）

目标：检出与基线逐检出一致（固定 9 图集）前提下最小化每帧延迟。基线 = 静态模式收口后：生产路径 93.3ms。

| iter | 改动 | 实测 | 判定 |
|---|---|---|---|
| 0 | 基线 + 仪表化（分段计时/VmHWM/--ot 旋钮/负样本图） | prod 93.3ms（load 39.0 / infer 43.0 / dec 0.9）；空帧出 `"dets":[]` | 基线 |
| 1 | `awnn_set_npu_ot` 带宽档位扫描 | MAXIMUM=43.0（=默认）· HIGHEST=43.4 · **LOWEST=103.0** | ✗ discard——**默认已是满档**；但证明模型带宽敏感 |
| 2 | 输出张量 CHW/FP32 → HWC/INT8（传输 2.45→0.61MB） | infer **不变** 43.0；**精度崩**：cls 量化 scale=10.45（背景 logit 极端负值撑爆动态范围）→ 分数只剩 0.5/1.0 两档、框吸附 8px 网格 | ✗ discard(revert)——**输出必须 FP32 是量化物理，不是习惯** |
