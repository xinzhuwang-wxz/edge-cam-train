# 观鸟器竞品地图 + 本项目端侧差异化定位

> 智能观鸟器/喂鸟器（Bird Buddy、Birdfy 等）这个**直接竞品品类**怎么做鸟类识别，以及本项目"端侧优先"的战略位置。两路竞品调研（Bird Buddy 深度 + 品类横扫）合成，2026-07-06。

## 二、品类"标准做法"（被功耗现实锁定）

**IPC 级 SoC 换壳的太阳能低功耗相机 + PIR 动检唤醒 + clip 上传 + 云端大模型分类 + 订阅变现。端侧只做动检和编码，不做鸟种分类。**
- **全行业清一色全云识别**：连标榜"AI Free Forever / 无订阅"的白牌机也强制要 WiFi（识别在厂商云）。**没有查到任何一款量产商用喂鸟器在片上跑鸟种分类**（唯一端侧案例是学术原型，且跑本地服务器非嵌入式）。
- **引擎三条路**：自研云（Bird Buddy/Birdfy）/ 调第三方大模型（FeatherSnap 用 Claude Haiku）/ 白牌共享识别 SaaS（中国低价机 10000–16000 种雷同）。
- **芯片**：喂鸟器 = IPC SoC，主流星宸 SigmaStar SSC33x(1–2T)、君正 T31/T40、全志 V831/V853、瑞芯微 RV1126。**这些 NPU 在喂鸟器里只用于 ISP/编码/动检，不承担分类**。
- **功耗现实**（3W 太阳能 + 电池、深睡+PIR 唤醒）把重活推到云：端侧留电给待机/唤醒/编码/上传。
- **种数军备竞赛** 6000–16000（云大模型才撑得起），实测准确率 70–90%，幼鸟/相似种/怪角度公认短板。
- **地域先验是质量分水岭**：好厂（Birdfy）用地理+季节加权，白牌（Harymor）不用、常推本地不存在的种。

## 三、Bird Buddy 深拆（头号竞品）

- **哑终端 + 全云**：5MP 传感器 + WiFi，无 NPU，断网即废，识别 100% 云端。
- **级联三段式**（与本项目几乎一样）：① 图像质量筛选（有鸟/对焦/眼喙可见）→ ② 检测器 crop → ③ 物种分类器。**差别只在整条链在云端、无区域先验、无层级回退**。
- **双档模型**：基础模型（免费自动）+ 最强模型 AI+（付费自动/免费手动）——"便宜模型兜底、贵模型做付费高精度"。
- **认不准 = "Mystery visitor" + 社区众包**，种 or Mystery 二元，无属/科回退。
- **护城河 = 数据飞轮**：硬件采集端 + 用户众包标注（确认/纠错/Mystery 众包，已处理 1300+ 纠正）→ 海量真实喂鸟器域内数据回流。
- **UX/变现**：Postcards→Collections 集邮式收集 + **Name That Bird 个体命名/回访**（"Bob is back"情感叙事）；识别质量做成付费墙。

## 四、本项目战略定位：标准硬件底座上的非标架构

- **随大流的部分**：V861 这颗 IPC-SoC 选型完全是行业标配（同族 V831/V853、星宸、君正满街都是）；两段式"粗检测→crop→细分类"算法结构也与竞品同构。**硬件选型零风险。**
- **差异化的部分（全行业没一家在做）**：把**细分类真正落到片上 NPU（INT8, 1 TOPS）**，而非上传云端。连"免费 AI"机都还在走云。
- **代价与自洽**：端侧 INT8 小模型撑不起 6000–16000 种军备竞赛——但这正好和本项目"高置信出片上标签、低置信/非地域/稀有种→层级回退(属/科/bird)+留云锚点"吻合。**本项目不打种数军备竞赛，打"当地几百种精准 + 分级回退 + 离线"**，分级门控恰恰是为兼容端侧算力上限而设计的。
- **价值 = 行业三大痛点的反面**：竞品"必须 WiFi / 识别在云 / 要订阅" → 本项目"**离线可用 / 隐私(图不离场) / 零云费 / 低延迟 / 省订阅**"。
- **识别智能领先**：区域先验（eBird seam）+ 层级回退（种→属→科→bird）+ 端云置信门控，是 Bird Buddy/多数竞品**没有的**（地域先验只有好厂用，层级回退无人做）。

## 五、必须应对 / 可抄的作业

1. **数据飞轮 ≠ 端侧劣势（2026-07-06 用户澄清，修正前误判）**：端侧做分类和"推卡片 + 收集用户确认/纠错"**互不冲突**——端侧分完照样推明信片、照样让用户确认/改正（同 Bird Buddy 众包），低置信样本可（经同意）选择性回传。全云=每张图都上云，端侧=默认不上云、**选择性回传高价值样本**（更精准+隐私可控）。**用户反馈闭环该建，但与分类在端/云无关**（应用层设计）。真正的数据难题是网络好图 vs 现场糊图的**域 gap**（靠自采 feeder + 回传补），不是"端侧收不到数据"。
2. **级联加"质量筛选"段（可抄）**：Bird Buddy 第一段判断有鸟/对焦/眼喙可见才进识别——即端侧最佳帧选择（IQA，OpenCV/Katna 可商用），省算力提准。
3. **UX 情感层（必抄）**：Collections 集邮式收集 + Name That Bird 个体命名/回访。识别是底座，留存靠这套。
4. **云端兜底可调 VLM（参考 FeatherSnap）**：云端最小兜底段可考虑调通用多模态大模型（Claude 等）做稀有种识别——但本项目端侧优先，这只是可选兜底。

## 六、对 roadmap / 命门的含义

- **端侧优先差异化确认**：不是随大流，是全行业没走的路；命门（层级可用率）正是为"端侧算力上限下仍给有用结果"服务，与定位自洽。
- **数据主线含"用户反馈回传闭环"**（低置信/纠错样本→云端重训→OTA）——任何路线都该建的数据飞轮，**与端/云无关**（[[classify-data-is-core-challenge]]）。
- **不追种数军备竞赛**：v1 端侧锁当地几百种（首发地域清单驱动），种数上限是特性不是缺陷（分级回退兜底）。
- **质量筛选段 + IQA** 可作 Round1/Round2 的 seam 预留。
