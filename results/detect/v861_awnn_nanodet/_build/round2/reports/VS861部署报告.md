## 1. 背景与源头（provenance）

- **源 ONNX** = `results/detect/round2/exports/main_416_fp32_logits.onnx`
  = round2 **main 模型**（NanoDet-Plus-m-416，1.0x，100% 数据；与 `mobile_handoff` 同一模型）的 **FP32 logits 导出**（按 ADR-0007 剥 sigmoid + 剥归一化）。
- **为什么用 logits 而非 mobile 的 `feeder_416.onnx`**：ADR-0007——检测 ONNX 出 logits、sigmoid/decode/NMS 留 CPU。而 `feeder_416` 把 sigmoid+归一化焊进图（移动端便利）。
  - 实测结论：AWNN **自动把 sigmoid 切到 CPU**（连 `feeder_416` 焊入的 sigmoid 也 Target=CPU）→ 两路 NPU backbone 完全相同、**不存在「sigmoid 进 INT8 图掉命门」**（回答了 ADR-0007 的待验证项）。

## 3. ★ 核心发现：检测头量化缺陷与混合精度修复

**症状（框级复现才暴露）**：INT8 整网 cos-sim 0.994 看着极好，但把参考输出喂 `decode_nanodet` 复现检测框：清晰 4 图里**松鼠漏检**、鸟分数被压。

**根因**：AWNN 默认全 INT8 + percentile 校准，**裁掉 cls 头稀有高峰值**（检测正是靠这些峰值过阈）。位移**不均匀、越高的峰压得越狠**：

| 目标 | FP32 顶分 | percentile INT8 | 掉 logit |
|---|---|---|---|
| 清晰松鼠（强检出）| **0.910** | **0.328**（漏检）| −3.03 |
| demo 鸟（命门）| 0.958 | 0.896（守住）| −0.98 |

**排错路径（一次只改一个变量）**：

| 量化策略 | 松鼠顶分 | 框召回(4图) | 结论 |
|---|---|---|---|
| 全 INT8 · percentile（默认）| 0.328 ❌ | 3/4 | 裁峰值 |
| 全 INT8 · minmax | 0.116 ❌ | 3/4 | **更差**（非校准算法问题）|
| **混合精度 · `gfl_cls` 头 fp32** | **0.458 ✅** | **4/4** | **修复** |

**定案**：**部署主配置 = `config_logits_mixed`（`hybrid_quantization_conf.white_list = /head/gfl_cls.0~3/Conv`）**，叠加 `use_npu_preprocess`（板端喂 0-255 BGR）= `config_logits_final`。代价几乎为零：`_ipu` 仍 1.3MB（头 conv 极小）。这与 ADR-0007「保护头」同向。

> **教训（写进方法论）**：**cos-sim 是差代理，检测任务必须做框级复现**。量化策略（混合精度/校准）是部署决策、直接决定板上精度——「部署=量化」。

## 4. 验证结果

**框级复现（mixed，4 张清晰图，INT8 vs FP32 同预处理输入）**：

| 图 | FP32 | INT8 | IoU |
|---|---|---|---|
| demo_bird | 1 | 1 | 0.971 |
| natural1_sparrow | 1 | 1 | 0.956 |
| natural3_blackbird | 1 | 1 | 0.972 |
| class_squirrel | 1 | 1（救回）| — |
| **合计** | 4 | **4** | 平均 **0.945** |

框召回 **1.000** / 框精度 **1.000**（零误报）/ 命门鸟框 IoU 0.95-0.99。

**交接包全面验收**（`test_board_handoff.py`）：**24/24 通过** —— 结构完整 + labels + config.txt 字段自洽 + 参考输出 decode 出正确 bird(0.850) + 交接包 INT8 ≈ FP32(IoU 0.971)。

**代码质量**：335 pytest 全过、ruff 全过。

**边界（诚实）**：ena24 camera-trap 的小/远目标（FP32 本身仅 0.52 勉强检出）INT8 仍会掉——**设计外**（观鸟器只看中大目标，见记忆 `feeder-scale-target-scope`）。

## 6. 阻塞项与盲点

- **真·上板**：真 INT8 延迟 / fps / 实测内存 / board-vs-sim 位匹配 —— **需 V861 板子**跑 `awnn_verify`。交接包已备好、一条命令即跑。**这是唯一未闭合的一步**。
- **模拟器 ≠ 真硅**：profile/simulate 是脱板估计，方向性可信；真数看板子。
- **校准可再优化**：当前 48 图小集（降温）出的 `_ipu` 是够用下界；机器凉/正式发布时可换回 282 全集（`calibration_datasets_full.txt`）重跑提精度。

## 7. 决策与文档指针

- **ADR-0009**：平台工具链更正（ACUITY→AWNN、.nb→_ipu、A7→RISC-V、V85x→V861）+ 部署路径重写。
- **ADR-0007**（沿用）：检测 ONNX 出 logits、sigmoid/decode/NMS 留 CPU。
- 混合精度头 fp32 = 本轮新增部署策略（记 docs/04 §4.1 + `AwnnPackager` 代码）。

## 8. 复现指南

```bash
# 1. 环境（一次）：加载 AWNN docker 镜像
docker load -i awnntools_v1.0.2.tar.gz        # 来自 vs861/AWNN-image.zip

# 2. 校准图：从 AutoDL 取 worksite 子集 → calib/worksite/（脚本见 git 历史）
#    或先用 mobile_handoff/demos 9 张冒烟

# 3. 转换量化（板端模型）：混合精度头 fp32 + npu 预处理
cd results/detect/v861_awnn_nanodet/_build/round2
./awnn.sh build   configs/config_logits_final.yml     # → build_out/logits_final/*_ipu.*
./awnn.sh profile configs/config_logits_final.yml out.csv
./awnn.sh sim     configs/config_logits_final.yml

# 4. 框级复现 + 交接包验收（本地，无 docker）
python decode_repro.py build_out/logits_final/results
python test_board_handoff.py

# 5. 框架调用（代码方式）
python -c "from edge_cam.deploy.packager import AwnnPackager; \
  AwnnPackager().pack('onnx/main_416_fp32_logits.onnx','out','/data/calib/calibration_datasets.txt')"

# 6. 板端（待 V861 板）：Tina SDK 编译 awnn_verify → 推 board_handoff/ → ./awnn_verify config.txt
```
