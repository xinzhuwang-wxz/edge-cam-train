- 07:03 监控器 v1 挂起；07:07 换 v2（+20min 心跳带 AP_50/bird）。4090D 17% GPU util = NanoDet 小模型 dataloader-bound，非异常。
- 07:27 心跳：ep11，bird 88.1%（ep10 val），健康。
- **08:27 main.done**（ep24 bird 91.8 / AP_50 0.858）→ sweep 开跑。
- **08:47 data12.5-1.0x.done**（val 峰 bird 84.3 / AP_50 0.667）→ 起 data25。
- **09:16 data25-1.0x.done**（val 峰 bird 87.1 / AP_50 0.726）→ 起 data50。
- **09:23 提速（后证误判）**：给 data100-0.5x/1.5x 把 `workers_per_gpu` 10→32，依据是"17% GPU util = dataloader 欠配"。
- **11:12 修正**：那个 17% 是**空档瞬时值**，训练非 dataloader-bound。容器 cgroup 配额 `1600000/100000` = **仅 16 核**（nproc 报 128 是宿主机）→ workers=32 **2× 超订** → 0.5x 反比 main 慢（0.75 vs 0.63 s/iter）。**已把 1.5x 改回 workers=10（与 main 一致）**；0.5x 跑到 ep12 改不动进程，接受半程慢 ~20%（约多 8min）。教训：AutoDL 容器看 cgroup 配额而非 nproc；单次 util 快照不足判 dataloader 瓶颈。
- **15:16 断线恢复**：本地会话崩退、Monitor 丢失；训练 screen `nd2exact`+`nd2fwdexact` 在 4090D 存活、**未受影响**。重连探针：A(1.5-exact) **ep17/24、bird 91.3**（ep10→17: 88.1→91.3，与 1.0x/B 同轨）、无报错、`sweep.done` 在。重挂持久 Monitor（epoch 进度/done/崩溃/卡死，`scratchpad/mon.sh`）。约 28min 收官。
- **15:50 data100-1.5x-exact.done (A)**：TRAIN_EXIT=0 干净收官，val 峰 bird 91.9 / final(ep24) 91.8 / AP_50 0.865。**A(91.9)≈B(91.6)≈1.0x(91.8) → 「neck warm-start 无关紧要、决策 B 无损」证实**（A full-COCO neck vs B 随机 neck，24ep 殊途同归）。全 7 点训练结束 → 进收口。

**test bird AP50 扫**（同一 test 集、横比公平；`test_eval/summary.txt` + 各 `*.test.log`）：
- **数据量曲线**（1.0x）：12.5→25→50→100% = **78.6 → 80.3 → 84.5 → 85.0**，单调升、边际递减，**50%→100% 仅 +0.5pt**（test 上比 val 饱和更早）。
- **参数曲线**（100%）：0.5x → 1.0x → 1.5x = **79.4 → 85.0 → 84.5(B)/85.1(A)**。0.5x 掉 5.6pt（容量不足）；1.5x 命门比 1.0x ±0.5pt（噪声内，不涨）→ **1.0x 甜点 test 上完全坐实**（对 0.5T 板子好消息）。

**结论（test 定案）**：部署模型 = **main（NanoDet-Plus-m-416, 1.0x, 100% 数据）**，test bird 85.0 / large 90.1。1.0x 是数据与参数双甜点；1.5x 不值、0.5x 不够。← scaling 双曲线 + 报告收口用此表。

**插跑 A（1.5-exact）**：full-COCO 拼权重（骨干 COCO-1.5x + neck/头 COCO-1.0x），加载实测 **fpn skip 3（B 是 175）**，仅 3 个 reduce 小卷积随机，≈1.0x 完整度。用途=**B vs A 量 neck warm-start 到底值多少**（B 已 91.6；若 A≈91.6 则证 neck 起点无关紧要、决策 B 无损）。首崩根因=拼权重丢 pl 元数据→`convert_old_model` KeyError，已修（保留完整 ckpt 结构、只换 backbone）。

**⚠️ warm-start 口径（影响参数 scaling 公平性，已处理）**：`gen_scaling.py` 原样复用主 config 的 `load_model`（1.0x COCO ckpt），导致 0.5x/1.5x 宽度不匹配→骨干加载不上。修正后各档 warm-start：
- **1.0x（主）**：COCO `nanodet-plus-m_416`（骨干+neck+头全 COCO 检测预训练）。
- **1.5x**：load_model = COCO `nanodet-plus-m-1.5x_416`。**实测（ep1 日志）：backbone skip=0（COCO 骨干全加载 ✅）+ gfl_reg/distribution 头加载 ✅，但 fpn(neck) skip=175 全随机**——因我们 config 用 **96-neck**（保宽度扫一致），官方 1.5x 是 **128-neck**，不匹配→neck 加载不上。故 1.5x = **COCO 骨干 + 随机→训练出的 neck**，非 full-COCO。
  - **决策 B（用户 2026-07-05，发现时已 ep18）**：**不重启**（重启白扔 45min，且 neck 起点差异 18ep 已基本抹平）。让其跑完，如实记此 warm-start 差异。对比仍成立（1.0x/1.5x 均 COCO 骨干）。备选 A（拼 full-COCO 权重重启）因 ep1 premise 已失效而弃。
- **0.5x**：**官方无 0.5x COCO 权重**（NanoDet-Plus 只发 1.0x/1.5x），只能 ImageNet `shufflenetv2_x0.5` 骨干 + COCO 96-neck/头（宽度无关部分从 1.0x ckpt 加载）。报告标注为**保守下界**（用户决定：照跑+说明）。
- **warm-start 汇总**（供报告诚实披露）：1.0x=full COCO；0.5x=ImageNet 骨干+COCO neck/头；1.5x=COCO 骨干+随机 neck。三者非完全对等，但**骨干（warm-start 大头）对 1.0x/1.5x 均为 COCO**；差异被 24ep 微调很大程度抹平。

**数据量 scaling 初读**（val 峰 bird_AP50）：12.5%→25%→50%→100% = 84.3 → 87.1 → 89.7 → 91.8，单调上升、边际递减，形状符合"数据够用、仍有微增益"。参数 scaling（0.5x/1.0x/1.5x）待两个 100% run 完成。**注**：val 峰是过程代理，最终以固定 test（test_test.json）评 model_best 为准（收口步）。

## ✅ 416 vs 320 分辨率消融结果（2026-07-05 完成）

**1.0x-320**（复用 main 的 416 COCO ckpt，input_size 320，24ep，数据/workers 与 416 对齐）val 峰 bird 90.9 / final 90.6。**固定 test 评测**（同 test_test.json）：

| 版本 | bird | squirrel | cat | person | other_animal | 整体 AP50 |
|---|---|---|---|---|---|---|
| **416 (main)** | **85.0** | 51.4 | 54.5 | 54.9 | 81.9 | 0.655 |
| **320** | **83.1** | 45.0 | 49.7 | 53.8 | 77.1 | 0.617 |
| Δ(320-416) | **-1.9** | -6.4 | -4.8 | -1.1 | -4.8 | -0.038 |

**结论**：降到 320 命门 bird 只掉 **1.9pt（85.0→83.1），仍稳超基线 77.4**；换来 ~1.7×（(416/320)²）省算力。**320 作为部署默认可行、守住命门**——对 0.5T/128MB 板子友好。非鸟类掉 ~5pt（本就弱环）。**部署建议：命门优先用 416（85.0/大目标90.1）；极致省算力可用 320（83.1）**。

## 后续：416 vs 320 分辨率小消融（用户 2026-07-05 定）

宽度扫（0.5/1.0/1.5x）全跑完后做：① 固定 test 评出 {0.5x,1.5x} 里**更优的一档**（bird AP50 为主，兼顾部署代价）；② 用 **1.0x + 那个更优档** 各训一个 **320** 版 → 对比各自 416 → 416 vs 320 分辨率消融。
- warm-start：320 版**复用各自的 416 COCO ckpt**（conv 权重与分辨率无关，架构同、直接加载）——同一份权重跑 416/320 两分辨率 → **分辨率成唯一变量，消融最干净**。（官方 320 ckpt 镜像下坏了且非必需，已弃。）
- 数据 100%、workers=10、24ep，与 416 版对齐；仅 `input_size` 320 vs 416 变。
- 动机：320 是部署默认（省算力、上板友好），验证降到 320 能否守住命门 bird 召回。**512 不做**（无官方权重 + 算力涨 ×1.5 + 只帮小目标而我们只做中大目标）。
